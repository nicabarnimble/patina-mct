Deep dive: standalone MCT versus patinaMother

This report compares the current patina-mct standalone product surface against the published patinaMother revision. It focuses on authority, child lifecycle, toys, peer transport, runtime execution, storage, observations, and operator workflows.

Standalone MCT patinaMother Authority boundary Runtime depth Code-read analysis

Start here

The new patina-mct product is now a credible standalone MVP for the Mother/Child/Toy shape: it has a typed kernel, a project-local child artifact loader, explicit child approval and assignment facts, first-class ToyGrant authority, JSONL observations, Iroh endpoint-ticket connectivity, a process harness, a long-lived process supervisor, a minimal Wasmtime component runtime, durable project-local config, peer address-book commands, and an Iroh serve path that can hand accepted peer calls to runtime execution.

Verdict: patina-mct is architecturally cleaner and much more explicit about authority than patinaMother, but patinaMother is still deeper operationally. The new product has the right shape for a standalone MCT node; it now needs hardening, not a return to the old broad ServerState model.

Method and evidence

I read the new MCT crates and the patinaMother code paths that correspond to child loading, child registry, daemon state, runtime host bindings, storage, toy grants, and operator workflows. I also ran the current workspace validation after the latest product work.

Product architecture: clean kernel versus broad daemon state

Standalone MCT

MCT is split into narrow crates: mct-kernel for domain facts and pure decisions, mct-observation for the JSONL ledger, mct-iroh for endpoint/protocol effects, and mct-daemon for CLI, config, child loading, process/WASM/toy adapters, and orchestration.

The kernel does not expose Wasmtime, Iroh, SQLite, process, or HTTP implementation types. That is the strongest architecture decision in the new product.

patinaMother

patinaMother has a broad ServerState in src/commands/mother/daemon.rs containing token, registry, runtime stores, services, scry backend, federation runtime, pando registry, native commands, aliases, refresh locks, warmup state, and readiness.

This breadth makes it productive but hard to reason about. The daemon state is simultaneously control plane, runtime registry, service container, pando orchestrator, and authority-adjacent state.

Authority deep dive

patinaMother already has important authority bones: users, nodes, visions, memberships, project identities, child registry entries, child installs, and project child assignments. Its SQLite schema even contains triggers that prevent active project assignments unless the registry entry is approved. The new MCT product did not copy this schema; it extracted the authority concepts into typed kernel records.

Authority fact MCT implementation patinaMother analogue Assessment
Peer admission MctPeerBinding, hello evaluation, call authority binding checks, Iroh unknown-peer denial. No Iroh peer authority found. MCT adds a new authority plane patinaMother lacks.
Child approval ChildApproval in kernel; durable MctStoredChildApproval in .mct/config.json. mother_child_registry_entries.state with states candidate, approved, blocked, deprecated. patinaMother has stronger relational durability; MCT has cleaner typed authority semantics.
Assignment ChildAssignment, active/revoked, scoped by Vision/Node/project. mother_project_child_assignments plus trigger requiring approved entry. MCT should eventually add database-level constraints or immutable fact replay equivalent.
Runtime instance ChildInstance with loading/ready/degraded/draining/stopped/failed; AuthorizedChildInvocation. Registry holds live Box<dyn Child> entries; readiness and warmup state are daemon projections. MCT's separation is cleaner. patinaMother has more live lifecycle behavior.
Toy authority ToyGrant, ToyGrantEvaluation, AuthorizedToyCall; manifest needs do not grant. GrantedToys boolean struct and host checks such as "sql toy not granted". MCT has the stronger authority model; patinaMother has much deeper host implementations.
The most important difference is not syntax. patinaMother often proves authority at the point of effect with booleans and store state. MCT proves authority by producing explicit, typed authorization records before the adapter can perform the effect.

Child lifecycle and registry

patinaMother's ChildRegistry is mature. It loads child entries into Arc<RwLock<Box<dyn Child>>>, stores wasm and manifest paths, maintains per-child reload locks, records typed call history, enforces ingress policy from manifests, emits metrics, runs drain/tick/task loops, and supports health checks, handle calls, typed WIT-shaped calls, and child swapping.

The new MCT registry is smaller and more declarative. It loads .wasm + .toml artifacts, computes SHA-256 digests, audits sidecars, parses ingress/allowlist/toy requests, detects duplicates, and can project loaded artifacts into kernel lifecycle records. It then gates route candidates through evaluate_child_call_authority.

Capability MCT status patinaMother status Next move
Artifact loading Good MVP. Digest and sidecar modes, manifest parse, duplicate denial. Mature. Hash verification, manifest engine parse, Wasmtime child instantiate. Keep MCT loader, add persistent artifact facts and install history.
Approval and assignment MVP. Durable JSON config and operator CLI. Mature SQL. Registry entries, installs, project assignments, triggers, audit. Promote MCT config to immutable facts or private SQLite with the same approved-entry invariant.
Ready instance Strong authority shape. Calls require ready authorized instance. Operational. Registry stores live children and readiness projections. Wire MCT ready instances to real process/WASM lifecycle and observation ledger persistence.
Reload and replacement Not deep yet. Kernel transition types exist; no replacement orchestration. Partial/mature. Reload locks and swap path exist. Implement replacement-load-before-swap in MCT supervisor/runtime.

Runtime execution

Process runtime in MCT

MctProcessChildHarness performs one-shot process calls only with AuthorizedChildInvocation. It writes JSON to stdin, collects stdout/stderr, handles timeout, returns MctResult, and emits runtime observations.

Long-lived supervisor in MCT

MctProcessSupervisor adds spawn/status/stop for long-lived process children, tracks pid/state/exit code, and emits runtime-started/completed observations.

WASM component runtime in MCT

MctWasmComponentRuntime uses Wasmtime component model to instantiate a real component and invoke a lifted s32 export only after child authorization.

patinaMother runtime

patinaMother has deeper WIT/WASI host bindings, child engine instantiation, state, events, SQL, git, measure, connect, and pando composition. It is much more complete as a runtime adapter.

MCT has crossed the important threshold from "artifact loader" to "real runtime execution". The remaining delta is WIT/WASI depth: payload conversion, generated bindings, host toy imports, traps, and persistent runtime observations.

Toys and host effects

patinaMother's toy host is broad and practical. The inspected code has host implementations that check GrantedToys booleans before SQL, messaging, measure, git, and other effects. That gives concrete host functionality, but authority is represented as a bundle of booleans carried inside runtime host state.

Security-significant asymmetry: the inspected patinaToy outgoing HTTP path uses host matching for credential injection, but unmatched hosts and requests without a host fall through to the default sender. That is allow-by-default egress. It must not be ported; a future network mctToy must be designed deny-unmatched from explicit ToyGrant scope.

MCT instead makes the grant itself a kernel object. A manifest needs list is only a request. Runtime toy execution requires an AuthorizedToyCall produced by pure grant evaluation. The daemon toy adapter registry currently has echo and failure backends and records ToyCallStarted, ToyCallCompleted, and ToyCallFailed.

Question MCT answer patinaMother answer
Can manifest needs grant authority? No. Tests prove missing grants deny. Manifest capabilities feed runtime grant decisions and host setup.
Can unknown toys run? No. Closed canonical toy catalog. Legacy alias bridge maps known names and denies unknown names; host booleans gate effects.
Are backend failures authority denials? No. They are toy adapter observations. Host errors are generally returned as runtime errors/strings and metrics.
Which is better? Better authority. Better backend surface.

Storage and observations

patinaMother's MotherRuntimeStore is a powerful SQLite store. It includes identity tables, guardrail triggers, child registry sources, child registry entries, installs, project assignments, audit tables, child state/checkpoints/subscriptions/offsets/tasks/runs, sessions, belief state, project registry, and more. This gives durability and relational invariants, but it also mixes MCT runtime state with Belief, session, scrape, and view concerns.

MCT intentionally starts narrower. JsonlObservationLedger is an append-only hash-chain ledger with identity and writer-lock checks. MctDaemonConfigStore persists child approvals, assignments, and peer address-book entries under .mct/config.json. This satisfies the standalone MVP requirement but is not yet a full state store.

Important product gaps after the MVP blocker pass

The user-listed blockers now have MVP implementations, but the product should not be called production-complete. These are the highest-value hardening gaps found in the comparison.

Gap Why it matters Recommended fix
CLI synthetic local authority process call, wasm call, and serve-process construct local approval/assignment facts for smoke execution. Route these commands through durable .mct/config.json approvals and child projections. Deny if not approved.
Observation persistence incomplete Many adapters return observations but do not always append them to the JSONL ledger. Make daemon runtime paths append before/after effect using JsonlObservationLedger.
WASM runtime is minimal Only lifted s32 export call is implemented; no WASI, WIT binding generation, host toys, payload conversion, or trap observations. Rebuild accepted WIT/WASI host requirements under MCT law, using patinaMother only as behavioral evidence and gating every effect with AuthorizedToyCall.
Supervisor has no restart recovery Long-lived process state is in memory. A daemon restart loses instance/supervisor state. Persist lifecycle snapshots and reconcile pids on daemon startup.
JSON config lacks relational guardrails patinaMother has SQL triggers that prevent active assignments to non-approved registry entries. Add invariant checks on load/save now; later move to private SQLite or immutable fact replay.
Peer address book lacks binding proof Endpoint tickets are reachability. They do not prove authority. Persist signed/issued MctPeerBinding facts with expiry/revocation and require them for call-peer.

Migration lessons from patinaMother

Translate and rebuild

SQL guardrails for approved child assignment, child registry source sync, install records, typed WIT allowlists, child warmup/reload, health/readiness, WASI host implementations, event/task state, and practical CLI ergonomics.

Do not copy wholesale

Broad ServerState, Belief/session/view tables in runtime state, stringly authority outcomes, host-effect booleans as the primary grant model, and ambient project/global path assumptions.

Translate

Translate patinaMother runtime concepts into MCT domain facts: registry entry to ComponentArtifact + ChildApproval, project assignment to ChildAssignment, host grant booleans to ToyGrant, runtime child slot to ChildInstance.

New MCT-only direction

Iroh endpoint tickets, peer bindings, MCT hello/call ALPNs, peer address book, runtime-routed Iroh replies, and observation-ledger truth as the main runtime audit surface.

Recommended next build sequence

Priority Slice Reason Proof gate
1 Config-backed runtime authorization Remove synthetic local CLI authority and prove durable approvals govern process/WASM/Iroh serve paths. Unapproved child fails; approved child executes; revocation stops or denies future execution.
2 Ledger persistence for runtime adapters Returned observations are not enough for runtime truth. Process, supervisor, WASM, toy, and Iroh serve paths append linked observations to ledger.
3 WIT/WASI host toy imports patinaMother's biggest practical advantage is WIT/WASI host depth. WASM child can call one toy import only through AuthorizedToyCall.

5 Private state store hardening JSON config is MVP. Production needs revisions, constraints, and replay. Invariant tests prevent active assignment without approved artifact; revocation has monotonic observation.

Bottom line

The new product is not a clone of patinaMother; it is a cleaner MCT kernel and adapter architecture informed by patinaMother. That is the right direction. patinaMother remains a source of operational requirements, especially WIT/WASI host bindings, registry SQL constraints, warmup/reload, and CLI workflows. MCT should retain its typed authority-first spine and rebuild accepted responsibilities as explicit domain facts, bounded adapters, and durable observations.

The next inflection point is config-backed end-to-end execution: durable approval + ready instance + peer binding + runtime adapter + ledger append. Once that is wired, MCT will be a standalone product in behavior, not only in crate structure.
