CTO brief: MCT P0 authority and ledger hardening

This report covers two scalpel commits in patina-mct: d5479e9 and b911e80. The executive read is straightforward: before expanding the runtime, the team closed two foundation risks. A peer call can no longer drift away from the authority granted during admission, and the observation ledger can no longer be reopened or appended as the wrong ledger without failing closed.

Audience: CTO Risk reduced Architecture invariant Operational guardrail No unsafe code added

Executive Summary

The core architecture rule remains intact: Mother kernel decides, adapters perform, and MctObservation proves what happened. This P0 batch reduces risk before the system grows more surface area. A successful peer hello is no longer treated as a broad permission; the call must match the exact binding, node, and Vision admitted by that hello. The ledger is no longer just a JSONL file with a valid hash chain; it must also match the intended ledger identity, Mother node, sequence, and single-writer boundary.

The management takeaway: this is not feature polish. It is risk retirement in the authority and audit substrate. These checks make later daemon, Iroh, child runtime, and toy-grant work safer to build because two core trust boundaries now fail closed.

Decision In One Sentence

This P0 batch converts two implicit trust assumptions into explicit typed checks: peer calls must match admitted authority, and ledger appends must match the current ledger identity and writer tip.

Why It Matters

Security posture

Calls cannot reuse a valid hello decision while changing binding, node, or Vision facts. That closes an authority-confusion class before multi-peer routing becomes more complex.

Audit integrity

The ledger now validates identity, sequence, and current tip before appending. A valid hash chain alone is no longer enough to accept a file as the current Mother ledger.

Delivery discipline

The work landed as two small commits with focused tests and clean workspace validation. That keeps the build path reviewable and reversible.

Change 1: call authority is no longer loose

Before this change, the call protocol checked that the referenced hello decision was admitted and that the endpoint and ALPN looked right. That was necessary, but too weak for an authority system. It did not prove that the call consumed the same peer binding, MCT node, and Vision that hello admission selected. For a CTO, the risk class is authority confusion: a real admission fact can be paired with different protected call facts unless the type path says otherwise.

Before

The call was attached to a hello decision, but the authority payload could name a different binding or Vision and still reach later checks. That is not a production-grade contract. 

if !hello.is_admitted()
    || request.authority.hello_decision_id != hello.decision_id
{
    return denied(... HelloNotAdmitted ...);
}

if request.authority.endpoint_id != request.received_over.endpoint_id {
    return denied(... EndpointMismatch ...);
}

After

The call must use the binding, node, and Vision from the admitted hello evaluation. A mismatch is a caller-safe denial, not an execution or adapter concern. 

if hello.selected_binding_id.as_ref()
    != Some(&request.authority.peer_binding_id)
{
    return denied(... BindingMismatch ...);
}

if hello.selected_node_id.as_ref()
    != Some(&request.call.caller.node_id)
{
    return denied(... CallerMismatch ...);
}

if hello.selected_vision_id.as_ref()
    != Some(&request.authority.vision_id)
    || request.authority.vision_id != request.call.caller.vision_id
{
    return denied(... VisionMismatch ...);
}
The principle is that the hello evaluation now behaves like a bounded authority token. The call protocol cannot widen it, reinterpret it, or pair it with different caller facts. That is the architecture we want before adding more adapter paths.

Change 2: expiry became an evaluation fact

MctPeerBinding already had expires_at, but the previous evaluation path only checked the explicit BindingState::Expired value. That left a valid-looking Admitted binding with a past timestamp dependent on some external projection to flip state first. The new shape treats expiry as an authority fact that the evaluation itself can decide. 

pub struct HelloEvaluationContext {
    pub ids: EvaluationIds,
    pub now: Timestamp,
}

if binding.expires_at.as_ref().is_some_and(|expires_at| expires_at <= &now) {
    return denied(
        request,
        ids,
        HelloReason::BindingExpired,
        SafeHelloReason::NotAuthorized,
    );
}

The context keeps the function pure and testable. It does not read wall clock time internally; callers provide the evaluation fact. That makes this logic easier to audit and less dependent on environment timing.

Change 3: ledger identity is part of the chain

A valid hash chain is not enough if the caller can reopen that file as a different ledger or Mother node. The scan path now validates ledger_id, mother_node_id, local sequence, and the hash chain together. The business value is audit integrity: the system rejects an internally consistent ledger file if it is not the right ledger for this Mother.

Before

let (next_sequence, previous_hash) = scan_existing(&path)?;
Ok(Self {
    path,
    ledger_id: ledger_id.into(),
    mother_node_id: mother_node_id.into(),
    next_sequence,
    previous_hash,
})

After

let ledger_id = ledger_id.into();
let mother_node_id = mother_node_id.into();
let lock_path = lock_path_for(&path);
let lock_file = acquire_writer_lock(&lock_path)?;
let (next_sequence, previous_hash) =
    scan_existing(&path, &ledger_id, &mother_node_id)?;

The new error variants are typed, not stringly: LedgerIdentityMismatch, SequenceMismatch, WriterLock, and LedgerChanged. That means operators and future control surfaces can distinguish wrong identity, broken sequence, lock contention, and stale writer state without parsing log prose.

Change 4: append is a named mutation boundary

The ledger now creates an exclusive sibling lock file such as observations.jsonl.lock. The open handle owns that writer boundary until drop. Before each append it scans the file tip again and refuses to write if the file changed behind the writer. This is not a distributed consensus mechanism; it is a deliberately narrow local writer contract for the current implementation slice. 

fn ensure_current_tip(&self) -> Result<()> {
    let (actual_sequence, actual_hash) =
        scan_existing(&self.path, &self.ledger_id, &self.mother_node_id)?;

    if actual_sequence != self.next_sequence || actual_hash != self.previous_hash {
        return Err(ObservationLedgerError::LedgerChanged {
            path: self.path.clone(),
            expected_sequence: self.next_sequence,
            actual_sequence,
        });
    }

    Ok(())
}
This is intentionally simple. It does not introduce a storage engine, background task, or speculative abstraction. It names the single-writer boundary and makes stale mutation fail before producing a new observation fact. That is the right risk-adjusted move before choosing heavier storage machinery.

Residual Risk

This is a P0 hardening pass, not the full runtime. The next risks remain persistent node identity and key lifecycle, production Iroh discovery and relay behavior, child lifecycle, toy grants, and durable storage beyond the JSONL proof ledger. The important progress is that the authority and append boundaries are now sharper before those surfaces expand.

The simplest next step is still to keep the system local, single-writer, and deny-by-default while one concrete daemon path grows around these invariants.

CTO Decision Points

Decision Recommendation Reason
Should the team add more runtime surfaces now? Yes, but only around the hardened boundaries. The authority token and ledger writer contract are now concrete enough to support the next daemon slice.
Should the JSONL ledger be replaced immediately? No. The current ledger is narrow, testable, and adequate for local-first proof work. Replace it when query, durability, or concurrency requirements demand it.
What should be de-risked next? Persistent node identity and daemon lifecycle. Those are the next authority-bearing surfaces. They should be built before child runtime or toy-grant complexity expands.
