Onboarding guide: MCT authority and ledger hardening

This is the educational version of the P0 hardening report. It explains the two commits d5479e9 and b911e80 for someone new to Rust, new to Patina MCT, or new to the way this project turns broad product goals into small, reviewable slices.

Audience: onboarding Rust vocabulary Project method No unsafe code added

Start here

MCT is an authority system before it is a networking system. That means the most important question is not "can two endpoints connect?" It is "who is allowed to do what, under which Mother, Vision, binding, and observation record?" The P0 work makes two of those answers explicit.

The project rule to remember is: Mother kernel decides, adapters perform, and MctObservation proves what happened. In Rust, we try to encode that rule in types and tests before adding more runtime behavior.

If you are new to Rust, read this as a guided tour of why we prefer explicit structs, enums, typed errors, small commits, and tests that prove denied paths, not just happy paths.

The Change In One Sentence

We made peer-call authority and ledger appends harder to misuse: a call must match the exact authority admitted during hello, and a ledger append must match the current ledger identity and writer tip.

Vocabulary First

Term Plain meaning Why it matters here
Mother The local authority runtime. It decides what is allowed. It does not blindly trust an adapter.
Adapter Code that talks to the outside world, such as Iroh networking or storage. Adapters perform effects. They do not create authority by themselves.
Vision An authority and sharing boundary. A peer admitted for one Vision cannot quietly act under another Vision.
Binding A typed record connecting an Iroh endpoint to MCT authority facts. Endpoint connectivity is not enough. The binding says what the endpoint is allowed to be.
Observation A structured fact about something that happened or was decided. The ledger is the runtime truth, so appends must be controlled.
Fail closed If the system cannot prove a request is authorized, it denies. This is the safe default for authority systems.

How We Build Here

This repository uses small implementation slices. A slice should have one concrete behavior, a small diff, tests, and a clear reason. The P0 work followed that pattern with two commits:

d5479e9

Tightened the kernel authority path. Calls now consume the exact binding, node, and Vision selected by hello admission.

b911e80

Tightened the ledger append path. The JSONL ledger now validates identity, sequence, hash-chain, writer ownership, and stale writer state.

Why two commits?

Kernel authority and ledger durability are different concerns. Small commits make review, rollback, and future debugging much easier.

This is the habit to copy: do not hide several architectural decisions in one large change. Make the behavioral boundary obvious, test it, then commit it.

Rust Idea 1: Make The Data Honest

Before the change, an admitted hello told us a peer was admitted, but it did not carry enough selected authority facts for the call path to check later. The fix was not a class hierarchy or a broad abstraction. It was to put the missing facts in the typed value.

Before

pub struct MctHelloAdmissionEvaluation {
    pub decision_id: DecisionId,
    pub request_id: String,
    pub selected_binding_id: Option<PeerBindingId>,
    pub hello_outcome: HelloOutcome,
    pub reason: HelloReason,
}

After

pub struct MctHelloAdmissionEvaluation {
    pub decision_id: DecisionId,
    pub request_id: String,
    pub selected_binding_id: Option<PeerBindingId>,
    pub selected_node_id: Option<MctNodeId>,
    pub selected_vision_id: Option<VisionId>,
    pub hello_outcome: HelloOutcome,
    pub reason: HelloReason,
}

Rust lesson: if later code must rely on a fact, make that fact part of a typed value. Do not make callers reconstruct it from memory, logs, or loose strings.

Rust Idea 2: Pass Context Explicitly

Expiry checks need to know "what time is it?" Instead of reading the clock inside the function, the new code passes time as part of HelloEvaluationContext. This keeps the authority function deterministic and easy to test. 

pub struct HelloEvaluationContext {
    pub ids: EvaluationIds,
    pub now: Timestamp,
}

if binding.expires_at.as_ref().is_some_and(|expires_at| expires_at <= &now) {
    return denied(
        request,
        ids,
        HelloReason::BindingExpired,
        SafeHelloReason::NotAuthorized,
    );
}
Rust design habit: prefer honest function inputs over hidden global state. That makes behavior easier to test and harder to misunderstand.

Rust Idea 3: Fail Before The Effect

The call protocol now checks binding, caller node, and Vision before the call can move toward protected routing. That is the right order. You do not want an adapter effect to happen and then later discover that the caller was not actually authorized. 

if hello.selected_binding_id.as_ref()
    != Some(&request.authority.peer_binding_id)
{
    return denied(... BindingMismatch ...);
}

if hello.selected_node_id.as_ref()
    != Some(&request.call.caller.node_id)
{
    return denied(... CallerMismatch ...);
}

if hello.selected_vision_id.as_ref()
    != Some(&request.authority.vision_id)
    || request.authority.vision_id != request.call.caller.vision_id
{
    return denied(... VisionMismatch ...);
}

Project lesson: the kernel denies before the adapter performs. This is why MCT separates authority decisions from external effects.

Ledger Lesson: A File Is Not Enough

The observation ledger is append-only JSONL with a hash chain. That is useful, but a hash chain alone does not prove the file belongs to the current ledger or Mother node. The P0 ledger change validates identity and protects the writer boundary.

Before

let (next_sequence, previous_hash) = scan_existing(&path)?;
Ok(Self {
    path,
    ledger_id: ledger_id.into(),
    mother_node_id: mother_node_id.into(),
    next_sequence,
    previous_hash,
})

After

let ledger_id = ledger_id.into();
let mother_node_id = mother_node_id.into();
let lock_path = lock_path_for(&path);
let lock_file = acquire_writer_lock(&lock_path)?;
let (next_sequence, previous_hash) =
    scan_existing(&path, &ledger_id, &mother_node_id)?;

Rust lesson: typed errors like LedgerIdentityMismatch, SequenceMismatch, WriterLock, and LedgerChanged tell callers what failed without parsing a string.

Tests Are The Proof

We do not only test success. In this project, denied paths are often the most important tests because they prove the system is safe by default.

Test What a new developer should learn from it
active_binding_past_expiry_is_denied Do not rely only on stored state labels when a timestamp also defines authority.
presented_node_claim_must_match_binding_scope Transport presentation is a claim; the binding scope is the authority fact.
presented_vision_claim_must_match_binding_scope Vision boundaries matter. Do not let a request drift across them.
call_authority_binding_must_match_admitted_hello A valid admission cannot be reused for a different binding.
call_caller_must_match_admitted_hello_node Caller identity must stay stable from admission to call evaluation.
reopening_with_wrong_ledger_identity_fails_closed A valid-looking file is rejected if it is not this ledger for this Mother.
second_open_writer_is_rejected Mutation boundaries should be named and protected.
append_fails_if_file_changes_behind_writer Stale local state must be detected before writing a new runtime fact.

How To Read The Diff

When you review a change like this, do not start by counting lines. Ask what invariant became more explicit.

Look for new facts

selected_node_id, selected_vision_id, and HelloEvaluationContext are not random fields. They are facts later code needs in order to be honest.

Look for denied paths

New reason variants such as BindingMismatch and VisionMismatch are signs that the system now names specific failure modes instead of hiding them behind generic errors.

Look for mutation boundaries

The ledger lock file and current-tip check define who may append. That is more important than the exact storage format at this stage.

Run tests

cargo test --all-targets

Unsafe code

No unsafe code was introduced. The crates keep #![forbid(unsafe_code)].

What This Does Not Finish

This hardening pass does not make MCT production-complete. Persistent node identity, production Iroh discovery, child lifecycle, toy grants, and more durable storage are still future work. The point of this slice is that future work now has sharper authority and ledger boundaries to build on.

When you work on the next slice, keep the same discipline: make one behavior explicit, write denied-path tests, avoid speculative abstractions, run the checks, and commit the smallest coherent change.

New Developer Checklist

Before you change code What to ask
Read the public structs and enums first. What facts does the API make explicit, and what facts are still loose?
Find the denied path. If authority is missing, stale, or mismatched, where does the system fail closed?
Keep adapters out of authority. Is this code deciding, or only performing an effect and reporting facts?
Prefer typed errors. Can a caller distinguish the failure without parsing a message string?
Commit in slices. Could this change be reviewed, tested, and reverted as one coherent step?
