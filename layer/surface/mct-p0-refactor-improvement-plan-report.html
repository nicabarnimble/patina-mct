Improvement plan: how I would refactor the P0 hardening

This fifth report explains how to improve on commits d5479e9 and b911e80 without pretending those commits caused every problem they exposed. The P0 work was a good local hardening pass. The next pass should turn its checked facts into harder-to-forge types, repair root assumptions that predate P0, and make the blast radius explicit before code changes begin.

Audience: four-way read Concrete code changes Blast radius Root rot called out No edits to prior reports

One Sentence

Keep the P0 checks, but refactor the surrounding model so authority, time, ledger entries, and public construction paths cannot be forged, half-populated, or overinterpreted by callers.

The root problem is older than the P0 commits: the crate began with broad public records, stringly timestamps, inert signature metadata, and wildcard exports. P0 added useful checks inside that shape; the next refactor should change the shape.

Refactor 1: Unforgeable Admitted Authority

The P0 call checks in call/internal.rs are good, but they are still checking two caller-supplied public values against each other. The next step is to stop accepting a raw MctCallProtocolAuthority as if it were an authority token.

Current Shape

pub struct MctCallProtocolAuthority {
    pub hello_decision_id: DecisionId,
    pub peer_binding_id: PeerBindingId,
    pub vision_id: VisionId,
    pub accepted_alpn: String,
    pub endpoint_id: EndpointIdText,
    pub policy_revision: u64,
    pub grants_revision: u64,
}

pub struct MctCallProtocolRequest {
    pub authority: MctCallProtocolAuthority,
    pub received_over: IrohConnectionPresentation,
    pub call: MctCall,
    ...
}

Target Shape

pub struct AdmittedCallAuthority {
    hello_decision_id: DecisionId,
    peer_binding_id: PeerBindingId,
    node_id: MctNodeId,
    vision_id: VisionId,
    endpoint_id: EndpointIdText,
    accepted_alpn: AcceptedCallAlpn,
    policy_revision: PolicyRevision,
    grants_revision: GrantsRevision,
}

impl AdmittedCallAuthority {
    pub(crate) fn from_admitted_hello(hello: &AdmittedHello) -> Self { ... }
}

Change evaluate_call_protocol to take an &AdmittedCallAuthority generated by hello evaluation, not an arbitrary authority field embedded in the wire request. The wire request can still carry authority claims, but the kernel should compare them to a verified local authority value.

Why this is Jon-style: the type name says what has been verified, its fields are private, and only the kernel can construct it from an admitted hello. A caller cannot build a fake AdmittedCallAuthority.

Yes. Public request and authority records were convenient scaffolding. Yes. MctCallProtocolAuthority and MctCallProtocolRequest came from b3512ac feat(kernel): add mct domain records. P0 d5479e9 added checks at call/internal.rs, but the public construction shape already existed.

Refactor 2: Enum State For Hello Admission

MctHelloAdmissionEvaluation currently uses optional fields for selected authority facts. That is understandable because denied hellos have no selected binding, node, or Vision. But it also means the type can represent an impossible admitted state.

Current Shape

pub struct MctHelloAdmissionEvaluation {
    pub selected_binding_id: Option<PeerBindingId>,
    pub selected_node_id: Option<MctNodeId>,
    pub selected_vision_id: Option<VisionId>,
    pub hello_outcome: HelloOutcome,
    ...
}

Target Shape

pub enum HelloAdmissionEvaluation {
    Admitted(AdmittedHello),
    Denied(DeniedHello),
    RetryLater(DeniedHello),
    UpgradeRequired(DeniedHello),
}

pub struct AdmittedHello {
    pub decision_id: DecisionId,
    pub binding_id: PeerBindingId,
    pub node_id: MctNodeId,
    pub vision_id: VisionId,
    pub accepted_alpns: NonEmptyAlpns,
    ...
}

The old is_admitted() helper can remain during migration, but the call evaluator should pattern-match Admitted. That makes missing authority facts impossible on the accepted path.

Files Specific Edits Blast Radius

Refactor 3: Parsed Time

P0 added an explicit HelloEvaluationContext { now }, which is the right direction. The weak root is that Timestamp is still a string wrapper and expiry compares strings. 

pub struct Timestamp(String);

if binding.expires_at.as_ref().is_some_and(|expires_at| expires_at <= &now) {
    return denied(... BindingExpired ...);
}

Change Timestamp to validate at construction and store a parsed instant internally, while preserving serde as RFC3339 text at the boundary. If dependency cost matters, keep a parsed internal newtype with a strict constructor and use strings only at serialization.

This predates P0. The string timestamp shape came from b3512ac feat(kernel): add mct domain records. P0 made time explicit but did not repair the type.
Code To Alter Why Tests To Add
crates/mct-kernel/src/id.rs Make invalid timestamps unrepresentable or at least construction errors. Reject malformed timestamps, compare equivalent canonical instants, serialize roundtrip.
crates/mct-kernel/src/peer/internal.rs Expiry should compare instants, not strings. Expired, exactly-at-expiry, future expiry, and non-canonical input cases.

Refactor 4: Ledger Entries As Verified Values

P0 improved validation, but the persisted entry type is still a public struct with public fields including entry_hash. That invites callers to treat arbitrary in-memory entries as verified ledger entries.

Current Shape

pub struct MctObservationLedgerEntry {
    pub ledger_id: String,
    pub mother_node_id: String,
    pub local_sequence: u64,
    pub observation: MctObservation,
    pub previous_entry_hash: Option<String>,
    pub entry_hash: String,
    ...
}

Target Shape

pub struct VerifiedLedgerEntry {
    inner: PersistedLedgerEntry,
}

impl VerifiedLedgerEntry {
    pub fn sequence(&self) -> u64 { self.inner.local_sequence }
    pub fn observation(&self) -> &MctObservation { &self.inner.observation }
    pub fn entry_hash(&self) -> &LedgerEntryHash { &self.inner.entry_hash }
}

Keep the serde shape private to mct-observation. Return VerifiedLedgerEntry from append and read APIs. Use accessors to avoid exposing mutable-by-construction persisted facts.

Root Rot? Predates P0? Blast Radius
Yes. Public persisted entries were added as part of the first JSONL ledger shape. Yes. b7f9246 feat(observation): add append-only jsonl ledger. Medium. Tests and query callers change from field reads to accessors. Wire file format can remain stable.

Refactor 5: Writer Lock And Durability Boundaries

The P0 ledger lock is good enough for a proof slice, but the next version should make the lock contract explicit and test stale lock behavior. The current code creates the file before acquiring the lock, removes the lock file on drop, and does not encode who owns a lock. 

if !path.exists() {
    File::create(&path)?;
}

let lock_path = lock_path_for(&path);
let lock_file = acquire_writer_lock(&lock_path)?;

impl Drop for JsonlObservationLedger {
    fn drop(&mut self) {
        let _ = std::fs::remove_file(&self.lock_path);
    }
}

Change this to a named LedgerWriterLock type with explicit owner metadata and policy. Acquire the lock before creating the ledger file. Add a default fail-closed stale-lock policy; do not auto-delete stale locks without a caller choosing recovery mode.

Code To Alter Specific Change Blast Radius
mct-observation/src/lib.rs Add JsonlObservationLedger::open_with_options, LedgerWriterLock, and LockPolicy. Low for default callers if open delegates to default options.
Tests Add stale lock, lock owner mismatch, create-after-lock, externally appended valid entry, and corrupted trailing line cases. Medium. Better tests may expose current behavior as intentionally unsupported.

Refactor 6: Fix Lifecycle Naming

evaluate_call_protocol_internal returns AcceptedForRouting with reason ResultRecorded. That is not a correctness bug today, but it is semantic rot. It makes the call lifecycle sound further along than it is. 

MctCallProtocolEvaluation {
    outcome: CallProtocolOutcome::AcceptedForRouting,
    reason: CallProtocolReason::ResultRecorded,
    safe_message: "accepted for routing".into(),
    ...
}

Add CallProtocolReason::AcceptedForRouting or AuthorityAccepted. Reserve ResultRecorded for the point where an actual result is recorded.

This predates P0. The base call evaluator came from 103cfab feat(kernel): evaluate mct call protocol. P0 added more denial checks, but it inherited this lifecycle naming problem.

Refactor 7: Curate The Public API

The kernel crate says it owns domain records and decisions, but it exports every module with wildcard re-exports. That is useful while prototyping, but it makes all public structs feel stable and encourages direct field construction. 

pub mod call;
pub mod id;
pub mod observation;
pub mod peer;
pub mod route;

pub use call::*;
pub use id::*;
pub use observation::*;
pub use peer::*;
pub use route::*;

Move to curated exports and constructors. Keep modules public if needed, but publish a deliberate surface: IDs, request wire types, evaluation functions, verified authority types, and observation conversion functions. Consider a prelude module for tests and examples rather than making every type a top-level export.

Yes. The export shape is broad by default. Yes. Most wildcard exports came from b3512ac; route export from 784d6f. Medium. Internal imports and tests update. External API compatibility should be treated carefully.

Audience-Specific Why

Engineering

This is not a rewrite. It is a targeted type refactor around existing functions. The code to touch is mostly peer, call, and mct-observation. Tests should move with each behavioral boundary.

CTO

The highest leverage next step is unforgeable authority. It reduces security ambiguity before new runtime surfaces multiply the number of call paths.

Onboarding

The lesson is that public structs are easy to learn but easy to lie with. A verified type should be something the system gives you after a check, not something any caller can type into existence.

Reality Review

This plan converts the fourth report's caveats into work. It does not hide root rot: several weak assumptions came from the initial domain record and ledger commits, not from the P0 hardening commits.

Final Position

I would keep the P0 commits. They made the system better. I would not pile more features on top of them yet. The next slice should refactor the authority and ledger shapes so the checks are backed by types, not just comparisons between public records. If the roots are rotted, they are rotted in the initial public domain model, string timestamp model, signature placeholder, and public ledger entry shape. P0 exposed those roots; it did not create them.

The crisp standard: after the refactor, an admitted authority value and a verified ledger entry should be impossible for arbitrary callers to forge. Until then, MCT has good local checks, not a finished authority substrate.
