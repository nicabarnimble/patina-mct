# feat: Release discipline — version, package, upgrade, notes, and baselines

> MCT 0.2.0 is built, identified, packaged, inspected, upgraded, and measured as one evidence-bearing product without turning release discovery into update authority.

## Problem

R2 closed the six R1 security findings, but the repository still has four independently declared `0.1.0` crate versions, no maintained changelog, no release archive, no SBOM, no release checksums/signing path, no guided binary-upgrade composition, and no production performance record. The current supervisor deliberately binds exact executable bytes and requires `install --replace`; replacing a binary outside that law causes the next supervised boot to fail closed. The current acquisition implementation proves Child packages and therefore cannot be relabeled as daemon release evidence without falsely classifying Mother as a Child.

This slice establishes release mechanics around the existing authority and lifecycle paths. It does not create a trusted channel, background updater, or network fetcher. The operator supplies one local archive, sees its exact notes and provenance, and approves one exact release digest. The guided verb then invokes the already governed lifecycle operations rather than creating a second replacement implementation.

## Goals

1. Move all workspace crates in lockstep to `0.2.0`, with `mct-daemon` as the product-version source.
2. Produce one target-named GitHub-Releases-style archive with normalized metadata, checksums, SBOM, license, release notes, and build provenance.
3. Support `aarch64-apple-darwin` honestly, including ad-hoc signing now and an activation-ready notarization slot.
4. Reuse the acquisition adapter, digest, evidence-ordering, and immutable-publication machinery for daemon releases without creating a `ComponentArtifact` or Child authority fact.
5. Add an evidence-informed `mct-daemon upgrade <artifact>` composition over exact operator approval and existing supervisor replacement.
6. Keep the upgrade planner independent of source kind so R5 can add fetch behind the acquisition seam rather than redesign upgrade.
7. Generate maintained release notes from conventional history with PR links and embed the exact committed version section.
8. Prove the packaged binary through a real local install, the three-fixture runtime proof, upgrade, and uninstall preservation.
9. Capture production baseline measurements without converting first-release observations into SLOs.
10. Refresh release/operator documentation only after the evidence exists.

## Non-Goals

- No in-daemon HTTP, Git, OCI, Iroh, registry, or other network fetch.
- No egress Toy, new canonical Toy, credential attachment, or coupled-slot implementation.
- No pre-authorized publisher, key, source, channel, latest-version selection, background check, scheduled update, or automatic approval.
- No change to exact-artifact update-authority law and no transfer of approval by version, source, archive name, signing mode, or checksum similarity.
- No notarization execution in v0.2; only the exact credential-gated insertion point is implemented and tested closed.
- No Linux binary, systemd adapter, Linux signing implementation, Homebrew formula, package-manager channel, JVM SDK, launcher, or interface work.
- No automatic rollback. R3 retains prior immutable releases and prints exact rollback guidance after a failed replacement or health check.
- No hard latency, throughput, memory, startup, or fixture-resource SLO.
- No ledger retention, compaction, repair, deletion, export, or alternate release evidence ledger.
- No mutation of `patinaMother`, its launchd policy, Patina interface files, or real Patina application state.
- No edit to ratified Allium law unless implementation discovers a conflict and the operator opens a separate amendment gate.

## D1 Decisions — proposed for operator ratification

D1.1–D1.14 are one operator gate. Implementation does not begin until D1 is ratified; later semantic changes are recorded as D1.x amendments.

### D1.1 — One workspace version is the product version

The root workspace declares:

```toml
[workspace.package]
version = "0.2.0"
```

Every member replaces its literal version with `version.workspace = true`. `Cargo.lock` records all four local packages as `0.2.0`. `mct_daemon::version()`, `mct-daemon version`, help output, release manifest, changelog section, archive name, and exact release tag all derive from that value. No second product-version file is introduced.

The tag format is exactly `v<semver>`; R3's release tag is the annotated tag `v0.2.0`. Version `1.0.0` is reserved for the separately adjudicated into-the-wild GA claim. A `0.2.x` artifact is a pre-GA release even when its runtime proof and release mechanics pass.

`scripts/check-release-version.sh` is added to tier-0 and enforces:

1. one root workspace version and inheritance by every member;
2. the same local-package versions in `Cargo.lock`;
3. a `CHANGELOG.md` heading for that exact version;
4. no higher released changelog heading than the workspace version; and
5. when HEAD has an exact `v*` tag, the tag is annotated and equals `v<workspace-version>`.

Developer and pull-request trees are allowed to have no exact tag. The publishing build is not: it requires a clean tracked/untracked tree in a fresh checkout and exact annotated `v0.2.0` at HEAD. The local smoke mode applies every agreement check except tag presence so it can prove the release before tagging.

**Rationale:** The crate graph and product cannot drift while ordinary development remains testable before the release tag exists.

### D1.2 — The archive has one exact, target-described layout

The distribution basename is:

```text
mct-daemon-v0.2.0-aarch64-apple-darwin.tar.gz
```

It expands to one root directory with no sibling entries:

```text
mct-daemon-v0.2.0-aarch64-apple-darwin/
  payload/
    mct-daemon.app/
      Contents/Info.plist
      Contents/MacOS/mct-daemon
  RELEASE-MANIFEST.json
  RELEASE-NOTES.md
  SBOM.cdx.json
  FIXTURE-PROVENANCE.json
  LICENSE
  CHECKSUMS
```

The minimal `.app` is a target-adapter payload, not a GUI product. It exists now so the same payload can receive a Developer ID signature and a stapled notarization ticket later without changing the archive contract. `RELEASE-MANIFEST.json` names the target-specific executable relative path; common verification, acquisition, and upgrade code never hard-codes `.app`, `Contents`, launchd, or `/Applications`. A future Linux package uses the same archive metadata contract with a Linux adapter-selected payload path such as `payload/bin/mct-daemon`.

`RELEASE-MANIFEST.json` is canonical JSON and contains at least schema version, product, product version, target triple, source commit, source epoch, Rust/Cargo versions, lockfile SHA-256, executable relative path and SHA-256/BLAKE3, notes/SBOM/provenance digests, selected distribution license, signing mode, and package-format version. `LICENSE` is the exact repository MIT license, and the manifest records `distribution_license=MIT`; workspace crates may continue to offer the additional Apache-2.0 option while this product archive deliberately selects one valid branch of `MIT OR Apache-2.0`. Dependency licenses remain enumerated by the SBOM rather than copied into this product license. The manifest contains no build-host home path, username, credential, environment dump, or wall-clock value not derived from the tagged commit.

`CHECKSUMS` contains two sorted records for every regular member except itself:

```text
sha256 <lower-hex> <root-relative-path>
blake3 <lower-hex> <root-relative-path>
```

The release directory also contains archive sidecars:

```text
mct-daemon-v0.2.0-aarch64-apple-darwin.tar.gz.sha256
mct-daemon-v0.2.0-aarch64-apple-darwin.tar.gz.blake3
```

Each sidecar uses `<lower-hex><two spaces><archive-basename>\n`. Upgrade requires both; neither algorithm substitutes for the other.

Common packaging fixes member order, directory/file modes, uid/gid to zero, uname/gname to empty, and all mtimes plus the gzip header mtime to the source commit epoch. Publishing runs only from a fresh clean checkout of the exact tag. Local build/smoke creates a detached temporary worktree of HEAD and builds there, so expected session/belief artifacts or any other untracked developer file cannot enter or influence the package. It builds with `cargo build --release --locked --target aarch64-apple-darwin`; it never runs `cargo update`. `BUILD-INFO` is represented by the release manifest rather than a second overlapping text file. The exact source revision, `rust-toolchain.toml`, `rustc -Vv`, `cargo -V`, cargo-sbom version, git-cliff version, target, and signing mode are captured in the release transcript and manifest.

**Rationale:** A manifest-selected payload keeps package and upgrade logic platform-neutral while normalized metadata makes the release intentionally repeatable from the committed lockfile.

### D1.3 — CycloneDX 1.6 is the SBOM contract

R3 pins `cargo-sbom 0.10.0` and generates one workspace SBOM with:

```text
cargo sbom --output-format cyclone_dx_json_1_6
```

A deterministic repository-owned normalizer sets the document timestamp to the source epoch, derives the serial identifier from product/version/target/source commit, sorts components/dependencies/properties, and rejects duplicate package identities. It does not remove dependencies emitted by the tool. The SBOM is generated from the committed `Cargo.lock` and records all workspace members plus transitive packages; scope properties distinguish runtime, build, and test-only material where Cargo metadata permits that distinction.

The normalizer adds the three committed proof fixtures as explicit `scope=excluded` components because they are release-proof inputs but are not shipped runtime payload:

- `slate-manager@0.2.0`;
- `folder-watch-actor@0.1.0`; and
- `watch-null-sink@0.1.0`.

Each fixture component records exact byte size, SHA-256, BLAKE3, committed receipt path, upstream repository/commit/tag, build command/toolchain, and patch digest/classification where applicable. `FIXTURE-PROVENANCE.json` carries the same bounded fixture records for operators who do not parse CycloneDX. Generation fails if a receipt, current fixture byte, or recorded hash disagrees. Neither file claims that proof fixtures are distributed inside the archive.

**Rationale:** The SBOM describes both shipped Rust supply-chain inputs and the exact non-shipped bytes used to prove release behavior without confusing those fixture bytes with product payload.

### D1.4 — Platform and signing behavior are adapters

The supported R3 matrix is:

| Target | Build | Supervisor | Signing | Release status |
|---|---|---|---|---|
| `aarch64-apple-darwin` | implemented | existing `gui/<uid>` launchd adapter | ad-hoc codesign | supported 0.2.0 target |
| Linux target (triple requires its own gate) | named future | `SystemdUserSupervisorAdapter` slot | `LinuxReleaseSigningAdapter` slot | unavailable; no artifact emitted |

Common release orchestration accepts a target adapter that supplies payload layout, binary inspection, signing, signature verification, and optional notarization. Common archive, manifest, checksum, SBOM, notes, acquisition, approval, and upgrade planning contain no macOS path, command, domain, or bundle assumption.

For v0.2, the macOS adapter builds the minimal app bundle, then runs:

```text
codesign --force --sign - --timestamp=none <bundle>
codesign --verify --strict --verbose=2 <bundle>
```

Signing occurs before payload digests, internal checksums, and archive creation. The manifest says `signing_mode=adhoc`; release notes state that ad-hoc signing proves package consistency, not Apple/notarized publisher identity.

The named notarization insertion point is after target payload assembly and before final payload digests/checksums/archive creation. It is disabled unless all credential-shaped inputs are present and an explicit release mode selects it:

```text
MCT_APPLE_SIGNING_IDENTITY
MCT_NOTARYTOOL_KEYCHAIN_PROFILE
```

Activation replaces ad-hoc signing with hardened-runtime Developer ID signing, creates a temporary submission ZIP of the already-defined app bundle, executes `xcrun notarytool submit --keychain-profile ... --wait`, executes `xcrun stapler staple` and `xcrun stapler validate` on that bundle, then resumes the same checksum/archive steps. Credentials and tool output do not enter the archive or ledger. Partial credentials, a rejected submission, failed staple, or unavailable tool fails closed and emits no release archive. R3 tests only the closed-slot selection and command plan; it does not submit.

The Linux signing slot has the same pre-checksum position and fails `unsupported release signing adapter` until separately specified. No systemd or Linux signing behavior is simulated by the macOS adapter.

**Rationale:** Choosing a staplable payload shape now makes notarization an activation of an existing seam rather than a later packaging redesign.

### D1.5 — Daemon release evidence is not Child evidence

The current Child `ArtifactAcquisition` and `ComponentArtifact` entities remain unchanged. R3 reuses their bounded filesystem reader, one-shot effect-capability pattern, SHA-256/BLAKE3 helpers, append-before-effect ordering, hidden staging, immutable digest-addressed publication, and resident/offline writer ownership. It does not fill `claimed_child_name` with `mct-daemon`, synthesize a WIT export, or create a Child catalog row.

R3 names a daemon-specific immutable projection:

```text
DaemonReleaseArtifactV1 {
  schema_version                    # exactly 1
  release_artifact_id               # sha256:<archive SHA-256>
  product                           # exactly mct-daemon
  product_version                   # exactly release manifest version
  target_triple
  archive_size_bytes
  archive_sha256
  archive_blake3
  release_manifest_sha256
  executable_relative_path
  executable_sha256
  executable_blake3
  release_notes_sha256
  sbom_sha256
  fixture_provenance_sha256
  source_revision
  rust_toolchain
  signing_mode                      # adhoc in v0.2
  source_kind                       # operator_file in v0.2
  source_ref                        # credential-free canonical file URI
  acquisition_decision_id
  adapter_effect_authority_ref
  acquisition_observation_id
  verification_observation_id
  immutable_release_path
}
```

Before reading the archive, the authenticated direct command durably records one `OperatorPointedDaemonReleaseAcquisitionDecisionV1` bound to canonical source ref, expected archive identity when supplied, product, target, attempt, UID, policy revision, and deadline. It is consumed by one success or failure and cannot become standing source trust. The private filesystem effect capability is still source/action/attempt/revision/deadline scoped and consumed by the adapter. This is a specialization of existing direct operator and adapter-effect authority, not a release channel or new transferable update authority.

The immutable path is:

```text
<service-root>/releases/sha256/<archive-sha256>/
```

It contains the verified extraction and a copy of the two external archive checksum records. State stores one query projection; the canonical evidence is the observation chain. A same-archive reacquisition adds attempt evidence and resolves to the same immutable release. A same identifier with different facts, a digest-path collision, or changed extracted bytes fails closed. Uninstall preserves the complete releases directory, release projections, state, and observations.

Required observation order, using existing kinds, is:

1. `OperatorActionRecorded` for the exact release acquisition decision;
2. `AdapterEffectStarted` before archive/sidecar open;
3. `AdapterEffectCompleted` or `AdapterEffectFailed` for bounded acquisition;
4. `ArtifactVerified` or `ArtifactRejected` for archive, layout, checksum, manifest, target, SBOM, and signature verification;
5. storage adapter start/completion/failure around immutable publication; and
6. later upgrade lifecycle facts, still separate from acquisition and verification.

No normal report is emitted if required evidence cannot become durable. Raw archive contents, environment, credentials, unbounded tool errors, and release-note text do not enter observations.

**Rationale:** Shared mechanics avoid a second unsafe file pipeline; a distinct record prevents Mother release bytes from acquiring Child ontology or authority.

### D1.6 — Verification is closed, bounded, and complete before approval

The v0.2 release verifier names these limits:

```text
MCT_DAEMON_RELEASE_ARCHIVE_MAX_BYTES       = 256 MiB
MCT_DAEMON_RELEASE_EXTRACTED_MAX_BYTES     = 512 MiB
MCT_DAEMON_RELEASE_MAX_ENTRIES             = 32
MCT_DAEMON_RELEASE_METADATA_FILE_MAX_BYTES = 8 MiB
```

Before immutable visibility or approval, it requires both external archive sidecars, computes both archive digests while streaming, and verifies the archive ID. Extraction accepts only the exact one-root layout for the manifest-selected target. Absolute paths, `..`, duplicate paths, path aliases, sparse files, symlinks, hard links, devices, FIFOs, sockets, extended entry types, extra top-level entries, excessive count/size, and a payload path not named by the manifest are rejected.

The two external sidecars must be regular files in the archive's canonical parent directory; symlinked, escaping, or multiply resolved sidecars are rejected. The verifier then checks every internal checksum, rejects missing or extra regular members, checks manifest/version/target/source/toolchain shape, parses the CycloneDX document, validates fixture-provenance agreement, and invokes the selected platform signature verifier. Every operator-displayed metadata file must be bounded UTF-8 and reject NUL, ANSI escape, C0/C1 control characters other than line feed, and terminal-direction overrides before any display. It does **not** execute candidate bytes before exact approval. Package creation proves the locally built executable reports the workspace version before assembly; an acquired candidate's runtime version is checked only after exact approval, replacement, and supervised start during post-verification. Archive-name metadata is diagnostic only; manifest and bytes decide. Target mismatch fails before executable publication or lifecycle effects.

Ad-hoc signature verification does not establish publisher trust. The operator-pointed decision establishes the one-attempt source path, and exact digest approval governs replacement. R5 may add stronger signing/notarization facts, but it may not weaken this floor.

**Rationale:** Release archives are hostile structured input until every path, bound, digest, and target fact has passed.

### D1.7 — `upgrade` is an exact direct-operator lifecycle composition

The production surface is:

```text
mct-daemon upgrade <artifact-ref>
  [--root <absolute-path>]
  [--expected-digest sha256:<64-lower-hex>]
  [--approve-artifact sha256:<64-lower-hex>]
  [--json]
```

In v0.2, `<artifact-ref>` accepts an absolute path or canonical `file://` reference and resolves only to `source_kind=operator_file`. HTTP(S), Git, OCI, Iroh, relative ambiguous references, credentials, URI user-info, query, fragment, and unknown schemes fail before any network or source effect. `--expected-digest` is additive and, when supplied, is bound before read. `--approve-artifact` is an automation form of the exact interactive approval, not `--yes`.

The command requires an existing valid current supervisor record owned by the authenticated UID. The canonical path and BLAKE3 of the currently executing upgrade CLI must equal that record's `executable_path` and `executable_digest`; invoking `upgrade` from a checkout, stale release, symlink alias, or unrelated copy refuses with guidance to use the currently installed executable. The trusted current version is therefore `mct_daemon::version()` from the exact record-bound running CLI, not a claim read from candidate bytes. It acquires and verifies the candidate while the current resident remains running when possible. It then displays, before any stop/replacement effect:

- current product version, executable path, and executable BLAKE3;
- candidate release artifact id, product version, target, archive and executable digests;
- source kind and credential-free source ref;
- acquisition and verification observation ids;
- source revision, toolchain, SBOM digest, fixture-provenance digest, and signing mode;
- the complete bounded `RELEASE-NOTES.md`; and
- the exact lifecycle plan and rollback guidance.

Interactive mode requires the operator to type the complete `sha256:<archive-digest>`. JSON/non-interactive mode requires `--approve-artifact` with exact equality. EOF, empty input, aliases such as version or filename, wrong digest, or a broad confirmation records a denied/no-change lifecycle outcome and returns failure before stop, record revision, plist publication, or start. Acquisition evidence remains because reading and verification already occurred.

Exact approval is carried by an existing authenticated `OperatorActionRecorded` plus linked `LifecycleTransitionRecorded` upgrade-start fact. Its bounded canonical detail names the current and candidate identities and the exact release artifact id. It is one direct lifecycle attempt, not a persisted standing approval, `ChildApproval`, update channel, or permission for another digest.

After approval, generic orchestration invokes shared lifecycle functions in this order:

1. request and complete the existing clean `stop` path, or observe its stopped no-op;
2. call the existing `install --replace` implementation with `--executable` equal to the manifest-selected executable under the immutable release path;
3. call the existing `start` implementation;
4. poll the owner-authenticated resident status until the fixed 30-second post-verify deadline; the status projection gains bounded `product_version`, supervisor revision, and executable-digest fields derived by the resident from its exact governing record and compiled product version;
5. require `running=true`, `health=healthy`, `readiness=ready`, the expected product version, supervisor revision, and executable digest; and
6. append upgrade/post-verify completion before returning success.

The implementation extracts shared functions; it does not shell out recursively to textual CLI commands and does not duplicate stop, replace, or start policy. Existing predecessor validation, loaded-service refusal, record revision, plist digest binding, append barriers, and writer fencing remain authoritative.

The lifecycle chain includes a bounded version delta even for equal versions:

```text
<from-version>@blake3:<old-executable> -> <to-version>@sha256:<release-artifact>
```

Equal-version/different-digest replacement is allowed only through exact approval so the smoke test can prove version similarity transfers no authority. A lower semantic version is rejected by `upgrade`; manual rollback uses retained immutable evidence and the existing stop / `install --replace --executable` / start sequence. A higher pre-GA version is allowed. Build metadata cannot make a digest equivalent.

On success, human/JSON output repeats the version delta, release artifact id, new executable digest, supervisor revision, acquisition/approval/install/post-verify observation ids, notes section counts, and ready status. The full notes are not copied into the ledger.

**Rationale:** The convenience verb shortens an operator procedure while preserving every existing authority and evidence boundary.

### D1.8 — Failure does not become implicit rollback or hidden success

A failure before exact approval has no supervisor effect. A stop failure prevents replacement. An `install --replace` failure prevents start and reports the current discovered record state. A start or post-verify failure is an observed failed upgrade and never reports the candidate as healthy.

R3 does not automatically bless the predecessor again or perform an unobserved rollback. The prior immutable release is retained. Failure output names only safe exact remediation based on the validated predecessor record, for example:

```text
mct-daemon stop --root <root>
mct-daemon install --root <root> --replace --executable <retained-previous-executable>
mct-daemon start --root <root>
```

The rollback path is ordinary direct operator lifecycle action and requires all existing observations and digest checks. If the writer is fenced or predecessor evidence is invalid, guidance says to preserve state and stop; it does not provide a force flag. A failure after the old resident stops is honest downtime, not a reason to weaken replacement validation.

Upgrade reports are projections over durable facts:

```text
DaemonUpgradeReportV1 {
  attempt_id
  outcome                         # completed | denied | failed
  from_version
  from_executable_blake3
  to_version
  release_artifact_id
  to_executable_blake3
  source_kind
  acquisition_observation_id
```

### D1.9 — The upgrade planner is source-kind neutral; R5 owns fetch

The CLI edge resolves `<artifact-ref>` through an acquisition-source registry. The v0.2 registry contains exactly `OperatorFileDaemonReleaseSource`. That adapter returns a verified `DaemonReleaseArtifactV1`. From the approval display onward, the upgrade planner accepts only the verified record and immutable payload path; it cannot inspect a URL, filesystem source path, response header, credential, or adapter implementation.

The R5 insertion point is `NetworkArtifactAcquisitionAdapter`. It must produce the same verified record after separately gated deny-by-default egress Toy, connection, source, initiation, verification, and secret-reference authority. Adding it may extend accepted artifact-ref schemes and source evidence, but cannot change approval, stop/replace/start, post-verify, report, or rollback contracts.

R3 adds no dormant HTTP client, URL parser that opens connections, credential field, fetch flag, channel, latest lookup, or feature-gated network implementation. The Track 3 `NetworkArtifactAcquisitionAdapter` row remains `DEFERRED`.

**Rationale:** Source neutrality belongs at the verified-evidence seam, not in a speculative generic downloader.

### D1.10 — `CHANGELOG.md` is generated, reviewed, and committed

The `0.2.0` section covers the repository's first-release history through the release commit and uses exactly these user-facing groups:

| Conventional input | Release section |

Breaking changes remain in Changed and are visibly labeled. The generated section is reviewed and committed before tagging. **Rationale:** One committed changelog provides reviewable release communication while the archive carries only the relevant immutable version section.

### D1.11 — The local release smoke uses the distributed bytes

`scripts/release-local.sh smoke` runs only on `aarch64-apple-darwin`. It refuses rather than skips when the host target, GUI launchd domain, required tools, archive verification, or exact fixed service-label availability is unsuitable. It acquires an exclusive per-UID smoke lock and proves `gui/<uid>/io.patina.mct.mother` is unloaded before mutation. It never stops, edits, or probes `patinaMother` beyond the ordinary unrelated service-label distinction.

The smoke extracts the verified archive into a temporary root and invokes the packaged app-bundle executable. It points `HOME` and `--root` at isolated owner-private temporary directories. Because launchd's fixed MCT label is global to the GUI domain, the smoke is serial and fail-closed rather than pretending the label is namespaced. A trap attempts only the smoke's current clean stop/uninstall; cleanup failure preserves the root and blocks release.

The required path is:

1. verify archive sidecars, layout, internal checksums, SBOM, signature, product version, and target;
2. run the packaged binary's real `install --root <temp> --executable <packaged-executable>` command, then real `start`, and await owner-authenticated UDS readiness;
3. copy the three committed proof fixtures into isolated source roots and perform operator-pointed acquisition through the packaged binary;
4. prove Slate denial before exact approval/ToyGrants, exact approval, scoped grants, and real Slate result;
5. prove temporal watcher invocation, scoped Watch delivery, ordinary exact sink call-out, and current evidence using the same behavioral steps as `supervised_trigger_watch_delivery_fixtures_execute_end_to_end`;
6. revoke the governed authority and prove denial survives one clean restart;
7. build a smoke-only second archive with the same `0.2.0` product/executable bytes but a different valid release-notes marker and therefore a different archive/release-artifact digest; it is marked `release_mode=smoke` and the publishing command refuses it;
8. run `upgrade` without/with wrong approval and prove acquisition evidence exists but supervisor record, running instance, and executable binding do not change;
9. run `upgrade --approve-artifact <exact-new-digest>`, prove record revision, clean stop, `install --replace`, start, ready post-verify, version-delta observations, notes display, and success summary;
10. cleanly stop/uninstall with the packaged successor binary; and
11. reopen ledger/state/releases and prove observations, fixture artifacts, authority projections, identities, runs, and both immutable release directories are preserved while the plist/current record/loaded service alone are absent.

The smoke captures stdout/stderr with secret redaction and command boundaries. It has no test-only production CLI flag selecting a fake supervisor, UID, label, launchctl binary, signature result, or approval result. Unit/integration tests may still use the existing private fake adapter for fault injection, but they cannot replace this close-out smoke.

**Rationale:** A source-tree test proves code; this smoke proves the bytes, layout, install path, and upgrade procedure an operator receives.

### D1.12 — Baselines are release evidence, not admission policy

The packaged, installed, ad-hoc-signed binary produces:

```text
layer/surface/build/product/BASELINES-v0.2.0-aarch64-apple-darwin.md
```

A target adapter records source revision, archive SHA-256/BLAKE3, executable digest, Rust/Cargo versions, OS version, architecture, hardware model, logical CPU count, memory size, power mode, sample counts, exact commands, and raw/summary values. It excludes username, home path, node secret, endpoint key, project contents, credentials, and unrelated process inventory.

The baseline harness records:

1. **Startup:** five clean launchd start/ready/stop cycles; monotonic duration from start request to owner-authenticated `ready`, with min/median/max and every sample.
2. **Idle RSS:** after 60 seconds ready and idle, seven 10-second-spaced RSS samples from the supervised PID, with min/median/max in bytes.
3. **UDS call latency:** 100 warmups plus 1,000 sequential calls to the already committed exact null-sink fixture, reporting p50/p95/p99/max, success count, and payload shape.
4. **UDS throughput:** four authenticated same-UID clients, 500 calls each, reporting wall time, calls/second, failures, resident CPU time, and peak RSS.
5. **Trigger-turn load:** a production-constant turn with 32 evaluation slots including the reserved terminal excess record, reporting turn wall/CPU time, admitted/terminal counts, ordinary status/call availability, and ledger growth.
6. **Three-fixture resources:** the complete fixture smoke segment under `/usr/bin/time -l`, reporting wall time, user/system CPU, peak RSS, archive/catalog/state/ledger byte deltas, and terminal outcomes.

Correctness failures, missing samples, wrong counts, or harness/tool errors fail the phase. Numeric values have no pass/fail thresholds in R3 and do not influence routing, readiness, or authority. Regressions may be discussed in the release checklist, but an SLO requires a later explicit gate.

**Rationale:** Captured methods and raw values establish a truthful first comparison point without pretending one machine defines production policy.
