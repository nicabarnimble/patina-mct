Reality review: P0 reports against the code

This fourth report fact-checks the three P0 authority and ledger reports against the actual commits, Rust sources, Patina context output, and current validation commands. The short verdict: the reports are materially accurate about the narrow P0 changes, but they sometimes compress local typed checks into language that can be misread as production security, durable storage, or cryptographic proof.

Start Here

The first three reports are not fiction. The commits they describe exist, the code paths are present, tests pass, clippy is clean, and the crates forbid unsafe code. The lies live mostly in the edges of wording: proves, runtime truth, foundation risks closed, and single-writer boundary can sound stronger than the current Rust and storage mechanics justify.

Honest version: P0 made two local invariants explicit. It did not finish MCT identity, distributed trust, production durability, public API tightening, or project-knowledge grounding.

MCT In General

The largest lie is not in any one report. It is the gap between the design-time ambition of MCT and the amount of runtime that currently exists. The project is building in the right direction, but the current implementation is still a local proof spine.

Identity

Endpoint and caller IDs are typed records. They are not yet backed by persistent key lifecycle, signatures, revocation, or replay-resistant admission tokens.

Storage

The ledger is useful local JSONL evidence. It is not yet a database, replicated log, crash recovery system, or tamper-evident signed audit store.

API Surface

Kernel records are mostly public structs with public fields, and mct-kernel re-exports broad modules. That is convenient now but not yet a tight stable API.

Knowledge Layer

Beliefs and generated context are not grounded enough. When patina context says there is no structure while assay sees code, the knowledge layer must be treated as advisory and stale.

Repair The Lies

Priority Repair Definition Of Done
P0 Ground or fix the Patina context/module inventory mismatch. patina context and patina assay inventory agree on crate/module existence, or context clearly labels structure as unavailable.
P0 Stop saying observations "prove" without qualifying the proof model. Future docs use "record," "evidence," or "audit fact" unless signatures, provenance, and verification are implemented.
P0 Implement or explicitly defer signed peer/caller authority. signature_ref is either verified through a concrete signature/key API or documented as inert metadata.
P1 Add ledger durability and concurrency tests that try to break the local writer contract. Tests cover stale lock, process crash simulation where practical, externally appended valid entry, corrupted trailing line, and concurrent open attempts.
P1 Tighten the public kernel API after the next slice proves which records must be stable. Wildcard re-exports are revisited, constructors enforce required invariants, and impossible states move out of public construction paths where it matters.
P2 Add property or fuzz tests for ledger hash-chain validation and call authority mismatch combinations. Randomized cases exercise sequence, hash, identity, binding, caller, Vision, ALPN, and payload mismatches without relying only on hand-picked examples.

Final Verdict

The three documents are accurate as audience-shaped reports for a narrow P0 slice. They become handwavy when their language is read outside that slice. The code really did move toward dependable Rust: explicit context, typed errors, fail-closed checks, and no unsafe. The current MCT runtime, however, remains a local proof spine with ungrounded project knowledge, unauthenticated identity metadata, a broad public API surface, and a JSONL ledger that is useful but not yet production durability.

The lie to prevent next is simple: do not let "typed local checks passed" become "authority system solved." The former is true and valuable. The latter is not true yet.
