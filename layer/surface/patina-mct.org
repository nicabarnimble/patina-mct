* Purpose

This document starts the slow design map for the new greenfield ~patina-mct~ product.

The integrated Patina repository contains ~patinaMother~, ~patinaChild~, and ~patinaToy~ prior art. It is *reference material*, not the source tree to move wholesale:

- Legacy integrated repo: [[https://github.com/NicabarNimble/patina/tree/d8f90270a53047b99d12004f834b62dbc629570d][patina integrated repo]]
- New MCT repo: [[https://github.com/nicabarnimble/patina-mct][patina-mct]]

The intent is to identify ~patinaMother~-shaped operational responsibilities, decide what ~mctMother~ must translate, rebuild, replace, or reject, and leave belief-domain meaning for the future Patina product operating as an ~mctChild~.

* Product boundary

** MCT owns

MCT is the Mother/Child/Toy runtime and private application fabric.

It should own:

- ~mctMother~ daemon lifecycle
- transports and control-plane API
- child registry and child lifecycle
- toy grants and capability policy
- WIT/WASI child runtime and toy capability model
- bridge/harness adapters: JVM bridge for banking infrastructure, process harnesses for protocol proof
- pando/application composition
- secrets/control-plane authority
- mandatory observability and view buffers
- HITL / AI interface launcher and control
- Iroh p2p identity and Mother-to-Mother transport as v0 direction; private-cloud/cluster policy later

** MCT does not own, initially

These belong to ~patina-belief~ or other products, and should only attach to MCT through an app/child/protocol boundary:

- ~scry~ retrieval internals
- ~oxidize~ embeddings
- ~assay~ structural knowledge queries
- scrape/index pipelines
- belief graph semantics
- Patina project database federation
- spec/lake/doctor builtins as hardcoded daemon internals

* Design stance

This is *not* an extraction checklist.

MCT must lock into the layer/core values:

- =dependable-rust=: small stable public APIs, private internals, one-sentence module jobs.
- =adapter-pattern=: traits at external-system edges only when the seam is real; honest signatures and domain types.
- =unix-philosophy=: compose focused modules/tools instead of a monolith.
- =safety-boundaries=: explicit authority, fail-closed behavior, no surprising side effects.
- =scalpel-commits=: build in small reviewable slices.

Instead:

1. Map the existing integrated features.
2. Decide whether each concept belongs in clean MCT.
3. Rebuild the minimal version in this repo only when the design is clear.
4. Use old code as reference, not cargo-cult source.
5. Apply the Jon Gjengset/Rust systems lens before implementation: typed seams, honest dependencies, explicit state machines, narrow public API, private internals.
6. Avoid boiling the ocean: Iroh v0, JVM bridge, WIT/WASI children/toys, observability, and Clojure-above-MCT must be modular slices, not one mega-build.

* Feature disposition answers

These rows should read as answers, not vibes. The discipline is:

- =Answer= says what ~mctMother~ would translate, rebuild, replace, or reject.
- =Agreement= says whether the answer is already agreed, proposed for discussion, or still needs study.
- =Evidence= links to the integrated repo surface that made the question concrete.
- =Reason= states the product-boundary logic in plain language.

Answer vocabulary:

- =rebuild= :: an accepted responsibility implemented intentionally under MCT law.
- =rebuild-minimal= :: an accepted responsibility, starting smaller than the ~patinaMother~ implementation.
- =defer= :: belongs in MCT, but not the first daemon slice.
- =bridge= :: attach through a child/app/protocol boundary rather than hardcoding into MCT.
- =belief= :: belongs to the future ~patina-belief~ product.
- =study= :: no answer yet; read and discuss before deciding.

Agreement vocabulary:

- =agreed= :: directly established in conversation or already accepted as a product boundary.
- =proposed= :: current answer offered for review.
- =study= :: intentionally unresolved.

** First narrow daemon slice

A narrow greenfield daemon could be:

#+begin_src text
mct-daemon v0
  local HTTP/JSON or UDS control plane
  Iroh node identity + production-shaped peer transport path
  /health
  /version
  child/component registry in memory
  process harness or simplest executable child path
  JVM bridge/client connection path
  child/component health
  child/component call
  append-only typed observation log
#+end_src

This proves the Mother/Child seam while making Iroh a real early substrate. It still keeps Wasmtime depth, full JVM orchestration, cluster policy, and the launcher out of the first slice.

** First runtime/bridge order answer

Status: =proposed=, with WIT/WASI child/toy center, Iroh Mother-to-Mother, JVM bridge, and Clojure-above-MCT direction now agreed.

| Order | Surface | Why this order |
|-------+---------+----------------|
| 1 | Tight edge-ready Rust kernel + local control plane | Proves the daemon seam without bloating the trusted core. |
| 2 | Iroh v0 identity/transport | Iroh is the Mother-to-Mother building system and must be a real early substrate, not a skeleton: identity, endpoint lifecycle, relay/discovery, peer admission, ALPN, direct streams, observations, and local multi-node tests. |
| 3 | Process harness or simplest executable child path | Proves lifecycle/routing/call semantics with minimal runtime friction. |
| 4 | JVM bridge/client path | Lets banking JVM infrastructure connect early into the WIT/WASI child/toy world without putting JVM internals in the kernel. |
| 5 | WASM/WIT/WASI child runtime | WASM/WIT/WASI is the child/toy design center; exact model evolves as we build. |
| 6 | Clojure interface/orchestration layer | Clojure sits above MCT as the developer/interface layer after the kernel is solid. |

* Modularity gates

Before any implementation slice, answer:

1. What is the one-sentence job of this module?
2. What public types/functions are stable enough to expose?
3. What stays private under =internal/= or equivalent?
4. Which external systems are touched, and are adapter traits justified by real seams?
5. What typed domain values replace raw strings at the kernel boundary?
6. What state machine or enum replaces boolean soup?
7. What observation is emitted for every authority decision?
8. Can the slice be committed and reverted independently?

* Open design questions

** Naming

- Is the executable ~mct~, ~patina-mct~, or eventually ~patina~ with product subcommands?
- Is “Mother” still the daemon name inside MCT?
- Does “AI launcher” become “interface launcher”, “guest launcher”, or “HITL launcher”?

** Protocol

- Start local control with HTTP/JSON, UDS, stdio JSON-RPC, or a combination?
- How much should MCT copy from Quack's pattern: familiar local interface, optimized substrate, typed semantic protocol, and remote capability feeling local?
- Where does HTTP belong exactly: JVM bridge, local tooling, Clojure interface, Mother-to-Mother fallback, or only as inspiration?
- What is the production-shaped Iroh v0 scope: direct streams + relay/discovery only, or also blobs/gossip/docs after reviewing the cached Iroh repo and companion protocol repos?
- Process harnesses, JVM bridge clients, WASM components, local HTTP/UDS, and remote peers should share one semantic call envelope.
- Remaining envelope question: exact Rust type names, payload codecs, and WIT operation encoding for the locked v1 shape recorded in [[file:../slate/work/mother-kernel-design/mother-kernel-design.md][mother-kernel-design]].

** State

- What is the smallest MCT state DB?
- Which ~patinaMother~ ~MotherRuntimeStore~ responsibilities belong in MCT versus Patina belief/session behavior?
- Should project identity be MCT-owned, Belief-owned, or shared contract?

** Launcher

- Which old ~patina ai~ behavior is essential in MCT v0?
- Should bundle projection and session artifact capture be part of first launcher slice or second?

** SDK boundary

- Which contracts must be authored in MCT first so SDK has a clean target?
